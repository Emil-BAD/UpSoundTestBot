# UpSoundTest

Небольшой, но Telegram-бота на `aiogram` для реального проекта.  
Бот умеет принимать ссылку на трек Яндекс Музыки и возвращать базовую информацию: название, исполнителя и длительность.

---

## 1) Название проекта и краткое описание

**UpSoundTest** — стартовая база под production-проект Telegram-бота с чистым разделением слоев:
- хендлеры только принимают/отдают сообщения;
- бизнес-логика живет в `services`;
- конфиг и запуск отделены от прикладного кода.

---

## 2) Функциональность

Сейчас в проекте есть:
- команда `/start` с русским интерфейсом;
- кнопка **«Трек»** с инструкцией;
- сценарий ожидания ссылки на трек;
- валидация ссылки Яндекс Музыки;
- получение данных трека через `yandex-music`:
  - название;
  - исполнитель;
  - длительность;
- обработка и логирование ошибок в терминал.

---

## 3) Используемые технологии

## 5) Установка и запуск

### 1. Создать и активировать виртуальное окружение

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
```

### 2. Установить зависимости

```powershell
pip install -r requirements.txt
```

### 3. Подготовить `.env`

Скопируй пример и заполни значения:

```powershell
copy .env.example .env
```

### 4. Запустить бота

```powershell
python main.py
```

## 7) Конфигурация и за что отвечают (`.env`)

Пример:

```env
BOT_TOKEN=your_telegram_bot_token_here
LOG_LEVEL=INFO
PROXY_URL=
YANDEX_MUSIC_TOKEN=
```

Что за что отвечает:
- `BOT_TOKEN` — токен Telegram-бота от BotFather (обязательно).
- `LOG_LEVEL` — уровень логов (`DEBUG`, `INFO`, `WARNING`, `ERROR`).
- `PROXY_URL` — прокси **только для Telegram** (aiogram-сессия).
- `YANDEX_MUSIC_TOKEN` — опциональный OAuth-токен Яндекс Музыки (fallback для некоторых API-сценариев).

Важно: внешние API-запросы (включая `yandex-music`) сейчас выполняются **без** `PROXY_URL`.

## 8) Кратко об архитектуре

Проект сделан по принципу «чисто, но без перегруза»:
- `handlers` — принимают апдейты, вызывают сервис, отправляют ответ;
- `services` — вся бизнес-логика и интеграции;
- `config` — загрузка настроек и `.env`;
- `utils` — общие функции;
- `middlewares` — прослойки между апдейтом и хендлером;
- `states` — FSM-состояния пользовательских сценариев.


---

## 9) Обработка ошибок (как происходит)

В проекте два уровня:

1. **Локально в сервисе**  
   `yandex_music_service` ловит ошибки API (невалидная ссылка, 404, сетевые ошибки, 451, таймаут и т.д.)  
   и возвращает понятный результат для пользователя.

2. **Глобально в боте**  
   `error_handlers.py` ловит необработанные исключения, пишет traceback в лог и не дает polling-циклу упасть.

В итоге бот не «молчит» и не падает в неожиданных местах.

## 10) Будущее масштабирование и что можно добавить

Что логично развивать дальше:
- подключить БД (PostgreSQL/Redis) для истории запросов и пользователей;
- добавить очередь задач под тяжелый парсинг;
- покрыть сервисы unit-тестами;
- добавить ретраи и circuit-breaker для внешних API;
- вынести интеграции в отдельные адаптеры (если появится несколько источников данных);
- добавить Docker и CI (lint + tests + smoke run).
